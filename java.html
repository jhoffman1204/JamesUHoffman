BuzzWord

What is BuzzWord?


My 2016 Fall Semester I programmed my first major project called BuzzWord. It's a game similar to "Boggle", where players must locate words on a grid to score points. Players locate the words by either dragging the mouse over the letters or by typing them as they are seen. The project highlights combining data structures, efficient gameplay algorithms and user interations.

The code can be found on my bitbucket account

https://bitbucket.org/JamesHoffman219/jameshoffmanbuzzword219

Data Structure

The UML diagram above shows the data structure for this project. I needed to store the information of different users, and that meant the levels they completed, the high scores for those levels etc. I store GameMode Objects into a HashMap of GameModes with the hash-key as the name of the Mode, and then each GameMode is initialzed with preset levels that can have their own time limits, points and personal bests.

The users are created as objects and then saved into json files using the ObjectMapper Class. Their information is stored in a local folder and be viewed and accessed by users trying to log in later on. The ObjectMapper class makes it extremely easy to save objects as shown below.

Full Grid Randomization

One of the challenges with this project was that the letters in the grid had to be randomized, but there also had to be a minimum amount of words that the player could find. Statistically, if you have a word bank of 50 countries and you randomize the letters in a 4x4 grid, it is almost impossible to get 3 words from the bank. If you repeatedly load and search the grid, the amount of operations would be too much, so there had to be a way to maintain randomness of the grid while not affecting the runtime of the program when an appropriate game board is needs to be loaded.

To solve this I created a method of inserting a specified length of word from any of my libraries into the grid, done through a depth first search algorithm. The idea was to randomly select a starting point for a word, then randomly select a direction for the next letters to go until the entire word was placed in the grid.

Depth First Search Algorithm



What Does This Accomplish?

1) Make sure that the random direction does not steer the word off the grid by throwing a checked excpetion if the coordinates of the random node does not exist.
2) Make sure that the word does not get trapped (by another word) so that the entire word cannot be placed. (Example: a 2x2 empty grid is available at the bottom left corner a 5 letter word is trapepd in that empty space). This is done through restarting the algorithm when a word has failed to be played too many times.
3) Checking to see that the random direction selected does not make one word run into another. This is the checkIfNullOnGameBoard method, which makes sure that both the tile does not already have a letter contained, and that the coordinates are a node on the game board.
4) Repeatedly check tiles surrounding until one is available. The algorithm will repeatedly try new directions until one direction is available. The counter in the four loop will decrement if the letter if the desired tile is not available, and so the letter the program is trying to place will remain the same.


Saving Profiles with Json Files

Json files are a way to store objects. They are most often seen in web development but java has a library called jackson that is used to save objects. I used the jackson library to save the profiles to a json file. It stores all the levels that were completed and the scores on those levels for each of the game modes. Json files are exremely simple to use with the jackson library. The picture above is a small representation of the entire json file, but you can see that individual objects are saved. The file can both be read later from the same ObjectMapper class, and the values that are put into the json file can be changed (if some are not required to be saved).
